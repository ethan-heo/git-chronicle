# 계획서(Plan) 작성 가이드 — GitChronicle

> **요약:** `docs/plans/` 임시 계획서의 표준 스켈레톤과 태그 기반 추적성 규칙을 정의한다. 새 계획서를 작성하거나 기존 계획서를 다른 세션에서 이어받을 때 참고한다.

> `docs/plans/`에 작성하는 임시 계획서의 구조를 정의한다. 계획서 자체의 존재 이유·폐기 규칙은 [documentation_guidelines.md의 "문서 최소화 원칙"](./documentation_guidelines.md#문서-최소화-원칙)을 따른다. 이 문서는 "무엇을 계획서에 남기는가"가 아니라 "계획서를 어떤 형태로 쓰면 다른 세션·다른 시점에서도 바로 이어받을 수 있는가"를 다룬다.
>
> 아래 표준 스켈레톤은 작업의 규모·종류와 무관하게 모든 계획서에 동일하게 적용한다. 작업별로 구조를 다르게 판단하지 않는다.

---

## 핵심 원칙 — 태그 기반 추적성

계획서가 다루는 변경 대상(UI 영역, 상태 값, 화면 등)마다 짧은 태그를 부여하고, 문서 전체에서 같은 태그를 반복 사용한다.

**예시** (사이드바+본문 레이아웃 계획서의 경우): 사이드바 3개 섹션 = `SB-1`/`SB-2`/`SB-3`, 본문 2개 섹션 = `MN-1`/`MN-2`(하위 `MN-2a`/`MN-2b`/`MN-2c`). 특정 영역에 속하지 않는 라우팅·전역 상태 변경은 `공통`으로 표기한다.

같은 태그가 아래 다섯 곳 전부에 등장해야 한다. 태그 하나가 이 중 한 곳에만 있고 나머지에 없다면 그 절이 비어있거나 놓친 항목이 있다는 뜻이다.

1. 청사진/와이어프레임 그림 안의 라벨 (표준 스켈레톤의 4.2, 4.3)
2. 변경 내역의 그룹 키 — 소제목 또는 표의 한 열 (표준 스켈레톤의 6)
3. Open Questions 각 항목의 접두사 (표준 스켈레톤의 8)
4. 구현 순서 체크리스트 각 단계의 접두사 (표준 스켈레톤의 9)
5. 수동 검증 체크리스트 각 항목의 접두사 (표준 스켈레톤의 10)

이렇게 하면 그림의 한 조각을 보고 태그를 확인한 뒤, 문서의 다른 절에서 같은 태그를 찾는 것만으로 관련된 모든 정보(무엇이 바뀌는지 / 뭐가 미결인지 / 구현이 끝났는지 / 어떻게 검증하는지)에 순서 없이 바로 접근할 수 있다.

## 표준 스켈레톤

```
# Plan: <제목>

> <표준 안내 문구 — 아래 "표준 문구" 참고>

## 상태
| 항목 | 값 |
|---|---|
| 단계 | 계획 확정 / 구현 중 / 검증 중 |
| 마지막 갱신 | YYYY-MM-DD |
| 구현 진행률 | n / m ("9. 구현 순서 제안" 참고) |
| 미결 Open Questions | n / m 미해결 ("8. Open Questions" 참고) |

## 1. 배경 및 목표
## 2. 사용자 확인 완료 사항        — 구두로 확정된 스코프 결정. 재논의 방지용으로 그대로 남긴다.
## 3. 현재 구조 요약               — 코드/문서 조사 결과. "왜 이렇게 바뀌는가"의 근거 스냅샷.
## 4. 신규 구조
   ### 4.1 정보 구조                — ID/이름 변경 등 식별자 수준 결정
   ### 4.2 레이아웃                 — 텍스트 트리, 태그 포함
   ### 4.3 와이어프레임              — 그림, 태그 포함
## 5. 상태 관리 / 데이터 변경안
## 6. 변경 내역 — 청사진 매핑        — 태그별 소제목 또는 표
## 7. 문서 갱신 대상                 — 문서 경로 | 태그 | 갱신 내용
## 8. Open Questions                — 번호(`n-1`, `n-2`...) + 태그 + 체크박스
## 9. 구현 순서 제안                 — 번호 + 태그 + 체크박스
## 10. 수동 검증 체크리스트          — 태그 + 체크박스
```

절 순서와 구성("변경 내역 → Open Questions → 구현 순서 → 검증"의 흐름 포함)은 모든 계획서에서 동일하게 유지한다.

## 표준 문구

새 계획서 맨 위에 그대로 복사해서 쓴다.

```markdown
> 이 문서는 임시 계획서다. 구현 완료 후 변경된 내용만 관련 spec/blueprint/project/core 문서에 반영하고 이 파일은 폐기한다. ([문서 작성 가이드](../project/documentation_guidelines.md) 참고)
>
> 이 파일은 완료 전까지 유일한 정본(single source of truth)이다. 다른 세션에서 이어받을 때는 이 문서 전체를 먼저 읽고, 상태와 체크박스만 보고 무엇이 끝났고 무엇이 남았는지 판단한다 — 별도 진행 상황 문서를 만들지 않는다.
```

---

## 체크박스 사용 규칙

- "8. Open Questions", "9. 구현 순서 제안", "10. 수동 검증 체크리스트"는 `- [ ]` 체크박스로 작성한다. 완료되면 `[x]`로 바꾸고, 결정 내용·실패 원인·스킵 사유를 항목 옆에 한 줄로 덧붙인다 — 체크만 하고 끝내지 않는다.
- "8. Open Questions"가 하나라도 미해결이면 "9. 구현 순서 제안"에 착수하지 않는다. 상단 "상태" 표의 "미결 Open Questions" 값으로 이를 한눈에 드러낸다.
- 순서를 건너뛰거나 바꿨다면 "9. 구현 순서 제안"의 해당 항목 옆에 이유를 짧게 남긴다.
